# APM導入 Phase 4: CI連携の検討

## 背景
APMの検証をCIへ組み込むことで、設定やlockfileの不整合を早期に検知できる可能性があります。一方で、外部依存取得や環境差分によりCIが不安定になる可能性もあります。

## スコープ
- `apm install`、`apm audit`、CI向けオプションの実行可否確認
- GitHub Actions追加の要否判断
- warning運用または必須チェック化の判断

## Phase 4の判断基準

### 必須チェック化の条件
- `apm install --dry-run` は、次のいずれかに該当する場合に必須チェック候補とします。
  - `apm.yml` に外部依存またはlockfileと整合すべき依存が追加された場合
  - テンプレート配布前に依存解決結果の再現性をCIで確認する必要がある場合
  - ローカル検証だけでは設定不備の見落としリスクが高い場合
- `apm audit` または `apm audit --ci` は、次のいずれかに該当する場合に必須チェック候補とします。
  - 外部依存、MCP、スクリプトなど、セキュリティ観点で継続監査が必要な要素を追加した場合
  - 監査結果の重大度に応じてリリースやマージを止める運用を決めた場合
  - `apm audit --ci` の終了コードと出力が安定しており、CIの失敗条件として扱えることを確認できた場合

### 失敗時の扱い
- APM CLIのインストール失敗は、CIの再現性に必要な前提が未整備な段階ではwarning扱いとし、CLIの導入手順とバージョン固定が決まった後に失敗扱いへ切り替えるか判断します。
- 外部依存取得失敗は、外部依存がテンプレート利用に必須でない段階ではwarning扱いとし、外部依存が成果物の正しさに直結する段階では失敗扱いを検討します。
- 必須チェック化する場合は、APM CLIの取得失敗、依存取得失敗、監査エラー、設定不整合をそれぞれwarning条件または失敗条件として明記します。

### 現時点の候補方針
- 現時点の `apm.yml` には外部依存がないため、まずは `apm install --dry-run` と `apm audit` / `apm audit --ci` をwarning運用で試行する方針を候補とします。
- warning運用では、CIへ組み込む場合もマージをブロックせず、実行時間、失敗頻度、出力の安定性を確認します。
- 外部依存を追加した時点で、必須チェック化の条件を再評価します。

## 判断結果
- CIへ組み込むかどうか: まずはwarning運用での組み込みを候補とし、必須チェック化は保留する
- 判断理由: 現時点の `apm.yml` に外部依存がなく、APM CLI導入手順と出力の安定性を先に確認する必要があるため
- 必須チェック化の対象候補: `apm install --dry-run`、`apm audit` または `apm audit --ci`
- 次の判断タイミング: APM CLIのCI導入手順が固まり、または `apm.yml` に外部依存を追加した時点
